
| **Term**          | **Definition**                                                                                                                                                                                                                                                                                                                    |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Deployment**    | A deployment is the high level object that manages a set of identical pods, and provides declarative updates to applications. This is used to automate the process of deploying, scaling, and updating containerized applications<br><br>“Deployment” is sometimes in lieu of the 'note' section on cluster definition.           |
| **Cluster**       | A cluster is a set of nodes across one or many VMs used to provide compute towards pods.<br><br>Note that "cluster" is also used to represent "the whole thing", including the nodes, pods, and services (as well as everything else). Nobody really uses the first definition provided very much.                                |
| **Node**          | A node is a separate virtual machine that acts as a “manager” for all your pods. A node has a fixed (or sometimes variable) set of resources (CPU/Memory/Disk) that it allocates to it’s pods.                                                                                                                                    |
| **Pod**           | A pod is the smallest single unit in Kubernetes. When you create your containers, the container(s) get bundled into a pod, which is what runs on the cluster. This pod will then execute your desired code, and make the cluster do something.                                                                                    |
| **Container**     | When you run your Dockerfile, your code, config files, virtual environment, ect … are all bundled into the container. You can imagine that if you sent this container to someone else’s computer, they would have all your virtual environments, dependencies, libraries, … letting you run code easily elsewhere                 |
| **Repo**          | A repository. This has nothing to do with Kubernetes, but is the location on GitHub/Lab where your code is located. It can be copied, written to, read from, ect …                                                                                                                                                                |
| **Service**       | A service is a stable set of IP addresses (with an DNS) that groups a set of pods, providing them a consistent way to communicate with each other without needing each pod to know each other’s ip address.                                                                                                                       |
| **Load Balancer** | A load balancer (conventionally) refers to a device that routes incoming traffic to different pods such that no one pod becomes overwhelmed by incoming traffic requests. This is typically done via a “round robin”, however other algorithms exist.                                                                             |
| **Namespace**     | A namespace is a rapid way of segregating "things", which allows for easy re-use of variable names. eg: if Smith and Jones have bank accounts, the fields under "namespace == Smith" will be the same as the fields under "namespace == Jones", but their two accounts have been quickly segregated such that they never interact |
